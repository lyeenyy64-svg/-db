# 2026년 3사 NPL 추심현황 — 데이터 구조 분석서

## Executive Summary

본 문서는 「★★★ 2026년 3사 npl 추심현황.xlsx」 파일의 전체 구조를 분석하여, 채권관리 앱 개발을 위한 데이터 모델링 기초자료로 활용하기 위해 작성되었다. 해당 파일은 3개 라스트마일 물류 브랜드(바로고·딜버·모아라인)의 NPL(부실채권) 추심 업무 전반을 Excel 기반으로 관리하는 통합 워크북으로, 총 21개 시트에 걸쳐 채권정보·입금관리·법적절차·분석 데이터가 분산 저장되어 있다. 앱 전환 시 핵심 엔티티 약 10개, 관계 테이블 5개 이상의 DB 설계가 필요하며, 특히 '채권관리' 시트(505행 × 789열)의 일별 추적 데이터 정규화가 최우선 과제이다.

## 2. 핵심 시트 상세 분석

### 2.1 채권관리 시트 (마스터 테이블)

#### 2.1.2 주요 데이터 패턴

- **브랜드 코드**: `B`(바로고), `D`(딜버), `M`(모아라인) — 1글자 약어 사용
- **채권 분류**: `장기채권`, `회생/파산`, `추심의뢰` 3가지가 주요 카테고리
- **추심 상태**: `추심진행`, `추심보류` 2가지 상태
- **코드 체계**: 숫자 또는 `숫자-서브번호` 형태 (예: `4134`, `4134-1`, `4134-2`) — 동일 채무자의 복수 채권을 하이픈으로 구분
- **대여일자 필드**: 날짜값 + 줄바꿈 후 "대위변제월" 텍스트가 혼재 — 파싱 시 주의 필요

### 2.2 입금내역 시트 (트랜잭션 테이블)

가장 정규화된 구조를 가진 시트로, 앱의 입금 트랜잭션 테이블에 직접 매핑 가능하다.

| 열 | 필드명 | 데이터 타입 | 설명 | 예시 |
|----|--------|-------------|------|------|
| C | 년월일 | Date | 입금일자 | `2026-01-01` |
| D | 구분 | String(1) | 브랜드 코드 | `B`, `M`, `D` |
| E | 담당 | String | 담당자 | `준원`, `덕진` |
| F | 허브/지점명 | String | 소속 허브/지점 | `충남천안원콜성두7지점` |
| G | 코드 | String/Number | 지점 코드 | `2172` |
| H | 채무자명 | String | 채무자 이름 | `한준희` |
| I | 입금자명 | String | 실제 입금자 | `김호순` (배우자 등 대리입금) |
| J | 계 | Number | 총 입금액 | `500000` |
| K | 본사계좌 | Number | 본사계좌 입금액 | `500000` |
| L | 캐쉬충전 | Number | 캐쉬충전 입금액 | `0` |
| M | 웰컴직접상환 | Number | 웰컴 직접상환액 | `0` |
| N | 비고 | String | 추가 메모 | `회생금` |

[분석] 입금 채널이 3가지(본사계좌, 캐쉬충전, 웰컴직접상환)로 구분되어 있으며, J열 = K + L + M 의 합산 관계

### 2.3 입금체크(BETA) 시트

좌측(C~G)은 일별 입금 체크 로그, 우측(K~T)은 회생 채무자의 월별 입금 체크 매트릭스로 구성되어 있다.

**[좌측: 일별 입금체크]**

| 열 | 필드명 | 설명 |
|----|--------|------|
| C | 날짜 | 입금 확인 대상 날짜 |
| D | 채무자명 | 채무자 이름 |
| E | 분할상환 메모 | 상환 계획 및 이행 메모 (여러 줄) |
| F | 입금액(누적) | 해당 건 누적 입금액 |
| G | 상태/잔액 | `완납`, `계획없음`, `🚨미납: XXX,XXX` |

[분석] 상태 필드에 이모지(🚨)를 사용한 시각적 알림 체계 운영 중 — 앱에서 알림/배지 기능으로 전환 가능

**[우측: 회생 입금체크 매트릭스]**

| 열 | 필드명 | 설명 |
|----|--------|------|
| K | 채무자명 | 회생 채무자 이름 |
| L~T | 1월~9월 | 월별 입금 금액 (0 = 미납) |

| L | 회차 | 현재 변제 회차 (예: `21회차`) |
| M | 월상환액 | 월별 상환 금액 |
| N | 변제 여부 | 변제 시작/종료 예정일 |
| O | 미납 여부 | 미납 상태 |
| P~Q | 진행 상황 | 상세 메모 |

[분석] 회차별로 월상환액이 변경되는 경우 **복수 행**으로 표현 (예: 1~36회차 → 146,783원, 37~60회차 → 다른 금액). 앱 설계 시 변제계획 테이블을 별도로 구성 필요

### 2.6 종합분석 시트 (1, 2)

주요 채무자에 대한 심층 분석 데이터를 **채무자별 열 배치**(피벗 형태)로 관리한다.

**[행 구조 — 분석 항목]**

| 행 카테고리 | 포함 항목 |
|-------------|-----------|
| 기본정보 | 종합분석일자, 허브, 생년월일, 내부 data, 지급당시 사업자 주소, 이메일, 연락처, 담당자, 이탈시기 |
| 최신정보 | 현재 사업자/자택 주소, 신용정보조회 주소, 현재 연락처, 현재 영업담당자 |
| 대여금 | 대여금 총금액, 잔액 |

## 4. 앱 설계 시 핵심 고려사항

### 4.1 데이터 정규화 우선순위

전체 워크북에서 가장 복잡한 구조를 가진 채권관리 시트의 정규화가 최우선 과제이다.

**[P0: 즉시 분리 필요]**

| 현재 구조 | 문제점 | 정규화 방안 |
|-----------|--------|-------------|
| 채권관리 AI~열 이후 (일별 추적) | 750+개 열이 날짜별 확장 → 열 무한 증가 | `추심활동로그` 테이블 분리 (채무자ID + 날짜 + 활동내용) |
| 채권관리 P열 (대여일자+대위변제월 혼재) | 날짜와 텍스트 혼합 | `대여일자` (Date) + `대위변제월` (String) 분리 |
| BD/M채권압류 (채무자 그룹핑) | 병합 셀 패턴, 채무자명 누락 행 다수 | `압류사건` 테이블 + `제3채무자_압류` 테이블 (1:N) |
| 분할상환 월별 열 (메모 텍스트) | 월별 열 증가 + 텍스트/금액 혼재 | `상환이행로그` 테이블 분리 (날짜 + 금액 + 메모) |

**[P1: 구조 개선 권장]**

| 현재 구조 | 정규화 방안 |
|-----------|-------------|
| 연대보증인 (콤마 구분 텍스트) | `채무자_연대보증인` 관계 테이블 |
| 연락처 (이력 포함 멀티라인 텍스트) | `연락처이력` 테이블 (전화번호 + 상태 + 변경일) |
| 코드 (숫자/하이픈 서브번호) | `코드`(지점) + `서브코드`(채권 건) 분리 |
| 종합분석 (피벗 형태) | 행/열 전환 → 채무자 프로필 + 분석노트 구조 |

## 5. 데이터 마이그레이션 유의사항

### 5.1 마이그레이션 순서 (제언)

앱으로의 데이터 이관 시 의존성을 고려한 순서이다.

1. **debtors** (채무자 마스터) ← 채권관리 시트의 B~Q열에서 추출, 중복 제거 및 ID 부여
2. **debt_accounts** ← 채권관리 R~AH열
3. **guarantors** ← 채권관리 F열 파싱 (콤마 분리)
4. **contacts** ← 채권관리 G열 파싱 (이력 분리)
5. **payments** ← 입금내역 시트 (가장 정규화된 데이터)
6. **collection_activities** ← 채권관리 AI열 이후 (일별 추적 → 행 전환)
7. **seizure_cases + seizure_targets** ← BD/M 채권압류 시트 (그룹핑 해제 필요)
8. **나머지 테이블** ← 각 시트에서 순차 이관

### 5.2 채무자 ID 생성 전략

현재 유일한 식별자가 없으므로, 다음 복합키를 활용하여 고유 ID를 생성해야 한다:

```
채무자 고유 식별 = 브랜드(B) + 코드(H) + 채무자명(E)
예: B-4134-㈜에스플러스 → debtor_001
    B-4134-1-㈜에스플러스 → debtor_002 (서브 채권)
```

[제언] 코드가 없는 경우(`코드없음`)가 다수 존재하므로, 마이그레이션 시 수동 ID 부여 또는 채무자명+브랜드+대여일자 조합을 보조 키로 활용
